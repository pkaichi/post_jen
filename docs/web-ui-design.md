# Web UI 設計

## 目的

postjen の操作と監視を行う Web UI を提供する。

現在の postjen は curl によるAPI操作のみに依存している。Web UI により、ジョブの登録・実行・監視・ログ閲覧を視覚的に行えるようにする。

## 基本方針

- UI は postjen-server とは別プロセスとして動作させる
- UI → サーバ間は既存の REST API を利用する
- リアルタイム更新は既存の SSE エンドポイントを活用する
- フレームワークは本設計では確定しない（実装フェーズで選定する）
- 実行基盤と UI の責務を分離する（implementation-policy.md の方針を踏襲）

## 画面構成

### 1. ダッシュボード

トップページ。システム全体の状態を俯瞰する。

主要要素:

- 直近の実行一覧（最新 N 件、状態バッジ付き）
- 実行中ジョブのリアルタイムステータス
- エージェント稼働状況サマリ（リモート実行対応後）

使用 API:

- `GET /api/runs?limit=N` — 直近の実行取得
- `GET /api/agents` — エージェント一覧（リモート対応後）

### 2. ジョブ一覧

登録済みジョブ定義の一覧表示と管理。

主要要素:

- ジョブ一覧テーブル（job_id, name, 有効/無効, 最終実行状態, 更新日時）
- ジョブ登録操作（YAML ファイルパス指定）
- ジョブ詳細へのリンク

使用 API:

- `GET /api/jobs` — ジョブ一覧取得
- `POST /api/jobs` — ジョブ登録

### 3. ジョブ詳細

個別ジョブの定義内容と実行履歴。

主要要素:

- ジョブ定義情報（name, description, definition_path, hash）
- ノード構成の DAG 可視化（ノード間の依存関係をグラフ表示）
- 実行履歴一覧（当該ジョブの過去 run を時系列表示）
- 実行開始ボタン

使用 API:

- `GET /api/jobs/:job_id` — ジョブ詳細
- `GET /api/runs?job_id=:job_id` — 当該ジョブの実行履歴
- `POST /api/jobs/:job_id/runs` — 実行開始

### 4. 実行詳細

個別 run の状態とノード実行状況。この画面が最も重要な監視画面となる。

主要要素:

- 実行メタ情報（run_id, status, trigger, 開始/終了時刻, 所要時間）
- ノード実行状況一覧（各ノードの状態、開始/終了時刻、exit_code）
- DAG ビュー上でのノード状態可視化（各ノードを状態に応じた色で表示）
- キャンセルボタン / 再実行ボタン
- SSE による自動更新（実行中の run はリアルタイムで状態反映）

使用 API:

- `GET /api/runs/:run_id` — 実行詳細
- `GET /api/runs/:run_id/events` — 状態遷移イベント
- `GET /api/runs/:run_id/stream` — SSE リアルタイムストリーム
- `POST /api/runs/:run_id/cancel` — キャンセル
- `POST /api/runs/:run_id/rerun` — 再実行

### 5. ログ表示

ノード単位の実行ログ閲覧。実行詳細画面から遷移または展開で表示する。

主要要素:

- stdout / stderr のストリーム別表示
- ログのページネーション
- 自動スクロール（実行中）
- system ログ（状態遷移メッセージ等）の表示

使用 API:

- `GET /api/runs/:run_id/logs?limit=N&offset=M` — ページネーション付きログ取得

### 6. エージェント一覧（リモート実行対応後）

リモートエージェントの状態監視。

主要要素:

- エージェント一覧テーブル（name, hostname, labels, status, last_heartbeat）
- オンライン / オフラインの視覚的区別
- 各エージェントで実行中のノード表示

使用 API:

- `GET /api/agents` — エージェント一覧
- `GET /api/agents/:agent_id` — エージェント詳細

## リアルタイム更新

既存の `GET /api/runs/:run_id/stream` は SSE で実行状態のスナップショットを返す。

UI での活用方針:

- 実行詳細画面を開いた際、run が終端状態でなければ SSE 接続を開始する
- SSE イベント受信時にノード状態・ジョブ状態を画面に反映する
- 接続断時はリトライする（EventSource の標準動作）
- ダッシュボードの実行一覧は通常のポーリング（10〜30 秒間隔）で更新する

## API 追加要件

既存 API で大半の画面を構成できるが、以下の追加・拡張が望ましい。

### 必要な拡張

| 内容 | 理由 |
|------|------|
| `GET /api/runs` に `job_id` フィルタ追加 | ジョブ詳細画面で当該ジョブの実行履歴を取得するため |
| `GET /api/runs` に `status` フィルタ追加 | ダッシュボードで実行中 run のみ取得するため |

### 将来的な追加候補

| 内容 | 理由 |
|------|------|
| `GET /api/stats` | ダッシュボードの集計表示（成功率、平均実行時間等） |
| `GET /api/jobs/:job_id/definition` | YAML 定義の生データ表示 |
| `DELETE /api/jobs/:job_id` | UI からのジョブ削除 |
| `PUT /api/jobs/:job_id` | ジョブ定義の更新（有効/無効切替等） |

## 画面遷移

```text
ダッシュボード
  ├─→ ジョブ一覧 ──→ ジョブ詳細 ──→ 実行詳細 ──→ ログ表示
  ├─→ 実行詳細（直近 run クリック）
  └─→ エージェント一覧（リモート対応後）
```

## 技術選定の方針

フレームワークは本設計では確定しないが、選定時に考慮すべき点を記載する。

- postjen-server の API はすべて JSON を返すため、任意のフロントエンドフレームワークで利用可能
- SSE 対応が必要（`EventSource` API を使うため、ブラウザネイティブで対応可能）
- DAG 可視化が必要なため、グラフ描画ライブラリが使えるフレームワークが望ましい
- postjen-server にバンドルするか独立デプロイするかは運用要件に応じて選択する
- SPA 構成を基本とし、API サーバとの通信のみで画面を構成する

## 段階的導入

### Phase 1: 最小限の監視画面

- ダッシュボード（実行一覧のみ）
- 実行詳細（ノード状態 + ログ表示）
- SSE によるリアルタイム更新

最小限の画面でも curl 操作の大部分を代替でき、実行状態の監視が格段に容易になる。

### Phase 2: ジョブ管理

- ジョブ一覧・詳細画面
- ジョブ登録・実行開始の UI 操作
- DAG 可視化

### Phase 3: 運用向け機能

- エージェント一覧（リモート実行対応後）
- 統計ダッシュボード
- ジョブ定義の有効/無効切替
